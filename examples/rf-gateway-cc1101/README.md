# 433 MHz RF Gateway with CC1101

> *Generated by [aurora@aurora-smart-home (esphome skill)](https://github.com/tonylofgren/aurora-smart-home)*

A sub-GHz RF gateway that exposes a CC1101 transceiver to Home Assistant
as a first-class `radio_frequency` entity using the new entity type from
ESPHome 2026.5.0. Send and receive arbitrary 433.92 MHz codes from HA
without per-device Python integrations: garage remotes, 433 MHz outlets,
old weather stations, generic doorbells, RF window contacts, you name
it.

## Features

- **HA-native RF control** through the 2026.5.0 `radio_frequency`
  entity (no custom Python or Bash-on-Pi)
- **Bidirectional**: receive new codes via `remote_receiver`, send any
  captured code or new code from HA
- **OOK and FSK modulations** advertised to HA via the ir_rf_proxy
  platform
- **Tunable frequency** (default 433.92 MHz, easily changed to 315 MHz,
  868 MHz, 915 MHz)
- **Standard ESPHome RF protocols** (NEC, RC5, Sony, Panasonic, raw)
  still work alongside the new entity type

## Hardware List

| Component | Purpose | Notes |
|-----------|---------|-------|
| ESP32 dev board | Host | Plain ESP32-WROOM-32 is fine |
| CC1101 module | Sub-GHz transceiver | E07-M1101D (TI CC1101 + amp) gives better range than a bare CC1101 |
| 17.3 cm wire antenna | Antenna | Whip antenna; tuned to 433 MHz quarter wave |
| Female-female jumper cables | SPI wiring | 6 wires: VCC, GND, CLK, MISO, MOSI, CS, plus GDO0 for IO |
| Optional 5 V to 3.3 V regulator | If powering CC1101 from a 5 V rail | The CC1101 is 3.3 V only |

The Sonoff RF Bridge R2 (CC1101 onboard) can be reflashed with this YAML
if you adjust the pins; that ships with a small case + antenna and
costs around 13 USD. See the `device-guides.md` reference for the pinout.

## Installation

### 1. Wire the CC1101

Follow the wiring diagram. Double-check VCC is 3.3 V, not 5 V. CC1101
modules are not 5 V tolerant on the SPI pins.

### 2. Solder a 17.3 cm wire antenna

A length of solid-core wire stripped of insulation works fine. Solder
it to the ANT pad on the CC1101 module. Range without a proper antenna
is poor (sub-1 m).

### 3. Flash

```bash
esphome run rf-gateway-cc1101.yaml
```

### 4. Capture codes from existing remotes

After flashing, open the ESPHome dashboard logs (or `esphome logs
rf-gateway-cc1101.yaml`). Point an existing 433 MHz remote (garage
opener, doorbell, light switch) at the gateway and press a button. The
log shows the captured timings:

```
[D][remote.raw:051]: Received Raw: 1000, -1000, 500, -500, ...
```

Copy those numbers. They are the codes you'll replay later.

### 5. Add automations and templates

Either:

- Add `button:` template entries in YAML that transmit captured codes
  (see the example button at the bottom of `rf-gateway-cc1101.yaml`)
- Or call `remote_transmitter.transmit_raw` from HA automations with
  the code embedded inline

## Configuration

### Different frequencies

```yaml
substitutions:
  frequency: "868MHz"    # for European LoRa-style devices
  # frequency: "315MHz"   # for US-region garage remotes
  # frequency: "915MHz"   # for US-region 915 ISM band
```

Also adjust the `radio_frequency:` platform's `frequency_min` and
`frequency_max` to bracket your target.

### Higher range with PA/LNA

The E07-M1101D module has a power amp and low-noise amplifier built
in. For long-range work (whole-property 433 MHz coverage) it's
roughly 4x the range of a bare CC1101. Costs about 8 USD vs 3 USD.

## Home Assistant Integration

### Entities Created

**radio_frequency**
- `radio_frequency.rf_gateway` - new 2026.5.0 entity type. HA targets
  this for any 433 MHz control via automations.

**Diagnostics**
- `sensor.rf_gateway_wifi_signal`
- `sensor.rf_gateway_uptime`
- `text_sensor.rf_gateway_esphome_version`
- `switch.rf_gateway_restart`

**Example button**
- `button.rf_gateway_send_doorbell_test` - placeholder, replace codes
  with your own captures

### Sending RF from automations

```yaml
action: remote_transmitter.transmit_raw
target:
  entity_id: button.rf_gateway_send_doorbell_test
data:
  code: [1000, -1000, 500, -500, 1000, -1000]
  carrier_frequency: 0Hz
```

Or use the new `radio_frequency.transmit` style if your HA version
supports it natively (works through the `on_control` trigger in
2026.5.0).

## Troubleshooting

| Issue | Likely cause | Fix |
|-------|-------------|-----|
| No codes received from a known remote | Wrong frequency or wrong modulation | Verify the remote's actual frequency (433.92 MHz is most common but not universal). Try OOK vs FSK. |
| Codes captured but replay does nothing | Timing tolerance too tight | Add `tolerance: 50%` and `filter: 250us` to `remote_receiver:`. Recapture. |
| Very short range | Missing or wrong-length antenna | Solder a 17.3 cm wire to ANT. Check it's straight and not coiled. |
| CC1101 not detected (SPI errors in log) | Wiring error or 5 V on a 3.3 V pin | Re-check VCC is 3.3 V. CC1101 modules are easy to fry with 5 V. |
| Frequency drifts between sessions | Cheap CC1101 clones with bad crystals | Buy from a reputable supplier. Real E07-M1101D modules are stable. |

## Why 2026.5.0 matters here

Before this release, exposing an RF chip to Home Assistant required:
- Either a custom Python integration on the HA side
- Or template buttons in ESPHome that send raw codes (no entity, no
  capability advertising)

The 2026.5.0 `radio_frequency` entity type plus the `ir_rf_proxy`
platform means HA finally has a standard way to talk to RF
transceivers. The entity advertises frequency range and modulations,
so any HA integration that knows how to use a `radio_frequency` entity
can target this gateway without knowing it's a CC1101 versus an RFM69
versus a custom chip.

The same wire-message family is shared with the existing IR proxy, so
the new entity type cost almost nothing in protobuf surface area.
