# Local Voice Assistant (ESP32-S3 + INMP441 + MAX98357A)

> *Generated by [aurora@aurora-smart-home (esphome skill)](https://github.com/tonylofgren/aurora-smart-home)*

A fully local, cloud-free voice assistant for Home Assistant. Wake word
runs on-device (micro_wake_word), speech-to-text uses the Whisper add-on,
the response is generated by your chosen conversation agent, and Piper
speaks it back through a real speaker. Nothing leaves your network.

## Features

- **100% local**: no Google, Alexa, or cloud STT/TTS - works offline once trained
- **On-device wake word**: "okay nabu" or "Hey Jarvis" via micro_wake_word, no streaming audio to HA until activated
- **Real speaker output**: MAX98357A class-D amp drives a 3W speaker, plenty for a kitchen or bedroom
- **Visible status**: WS2812 RGB ring shows idle / listening / processing / responding
- **HA Assist native**: plugs straight into the pipeline you already use on your phone
- **Cheap**: under $20 in parts, no PCB required
- **Open**: every line of firmware is yours to edit

## Hardware List

### Microcontroller

- **ESP32-S3-DevKitC-1-N16R8** (16 MB flash, 8 MB PSRAM, ~$10)
  - PSRAM is **mandatory** - micro_wake_word + voice_assistant will not fit in plain SRAM
  - The N8R2 variant works too; the cheaper N4R2 does not have enough flash for esp-idf

### Audio & Indicator

| Component | Purpose | Interface | Notes |
|-----------|---------|-----------|-------|
| INMP441 | I2S MEMS microphone | I2S in | Omnidirectional, 24-bit, very low noise floor |
| MAX98357A | I2S DAC + 3.2W class-D amp | I2S out | Tie GAIN to GND for 15 dB |
| 3W 4 ohm speaker | Audio output | Wired to MAX98357A | Any 4-8 ohm speaker works |
| WS2812 (single pixel) | Status LED | 1-wire | Optional but very useful for debugging |

### Wiring

The N16R8 has two free I2S controllers - use one for mic, one for speaker.

| Signal | ESP32-S3 GPIO | Module pin |
|--------|---------------|------------|
| **Mic WS / LRCLK** | GPIO 4 | INMP441 WS |
| **Mic SCK / BCLK** | GPIO 5 | INMP441 SCK |
| **Mic SD / DIN** | GPIO 6 | INMP441 SD |
| Mic L/R select | GND | INMP441 L/R (left channel) |
| Mic power | 3.3 V | INMP441 VDD |
| **Spk LRC** | GPIO 7 | MAX98357A LRC |
| **Spk BCLK** | GPIO 15 | MAX98357A BCLK |
| **Spk DIN** | GPIO 16 | MAX98357A DIN |
| Spk gain | GND | MAX98357A GAIN (15 dB) |
| Spk power | 5 V | MAX98357A Vin |
| **Status LED** | GPIO 48 | WS2812 DIN |

All grounds tied together. Power the whole puck off the DevKit USB-C jack.

## Software Requirements

Home Assistant must have these add-ons installed and a pipeline configured:

1. **Whisper** add-on (STT) - pick `tiny-int8` or `base-int8` for fast local response
2. **Piper** add-on (TTS) - the `en_US-lessac-medium` voice is a good default
3. **openWakeWord** add-on - optional; only needed if you also want HA-side wake word
4. **Settings > Voice assistants > Add assistant**
   - Conversation agent: Home Assistant (or any local LLM via Ollama / Extended OpenAI Conversation)
   - Speech-to-text: Whisper
   - Text-to-speech: Piper
   - Wake word: leave on "None" - the ESP handles it

## Build Steps

1. **Wire up the puck** on a breadboard following the table above. Double-check
   3.3 V on the INMP441 and 5 V on the MAX98357A.
2. **Copy `secrets.yaml.example` to `secrets.yaml`** and fill in WiFi, OTA, and API key.
3. **First flash over USB**:
   ```bash
   esphome run voice-assistant-s3-box.yaml
   ```
   The first esp-idf build takes 10-15 minutes. After that, OTA is the norm.
4. **Adopt in Home Assistant**: it appears as a discovered ESPHome device. Click Configure.
5. **Assign the pipeline**: Settings > Devices & Services > ESPHome > the new device > Configure > Voice assistant > pick the pipeline you made above.
6. **Test**: say "okay nabu, turn on the kitchen light". LED goes yellow, then green, then blue while Piper replies.

## LED States

| State | Color | Meaning |
|-------|-------|---------|
| Idle | Dim red | Listening for wake word |
| Listening | Yellow | Wake word detected, recording your command |
| Processing | Green | Whisper transcribed, intent running |
| Responding | Blue | Piper is speaking |
| Error | Bright red | Pipeline failure - check HA logs |

## Customization

- **Change the wake word**: edit `wake_word_model` in the substitutions block. Built-in choices are `okay_nabu`, `hey_jarvis`, `alexa`, `hey_mycroft`.
- **Train your own wake word**: use the [microWakeWord training Colab](https://github.com/kahrendt/microWakeWord) and host the resulting `.json` on a local HTTP server, then point `model:` at that URL.
- **Quieter wake-up**: drop `volume_multiplier` from 2.0 down to 1.0.
- **More aggressive noise suppression**: raise `noise_suppression_level` from 2 to 4 if the room is loud.

## Troubleshooting

- **"Component requires PSRAM" build error**: you have an N16R2 board without PSRAM. Switch to N16R8 or N8R2.
- **Wake word never triggers**: the INMP441 L/R pin must be tied to GND for left-channel data. Verify on a multimeter.
- **Audio is distorted or buzzing**: MAX98357A is power-hungry; feed it 5 V from a stable source, not the 3.3 V rail.
- **Speaker is too quiet**: tie MAX98357A `GAIN` to GND (15 dB) or leave floating (9 dB). 100k pulldown to Vin gives 3 dB only.
- **Long delay before response**: switch Whisper to `tiny-int8` and Piper to `low` quality. Expect ~1.5 s round trip on a Pi 5.
- **OTA fails after first flash**: the esp-idf binary is large; make sure your YAML is on the same WiFi as HA and OTA isn't blocked by firewall.

## Why ESP32-S3 (not S2 or original ESP32)?

- micro_wake_word needs AI vector instructions, which only S3 has.
- voice_assistant needs >2 MB of free RAM for the audio ring buffers; PSRAM solves that.
- The original ESP32 works in HA voice projects but only as a "press to talk" puck with no on-device wake word.

## Simpler Alternative: M5Stack ATOM Echo

If you want to skip the breadboard, the **M5Stack ATOM Echo** is a tiny pre-built puck with mic and speaker built in. Set `board: m5stack-atom` and remove the I2S pin assignments; ESPHome ships a ready ATOM Echo example. The trade-off is no PSRAM, so wake word runs on the HA side via openWakeWord instead of on-device.
