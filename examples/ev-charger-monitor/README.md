# EV Charger Monitor Example

> *Generated by [aurora@aurora-smart-home (esphome skill)](https://github.com/tonylofgren/aurora-smart-home)*

Polls an OpenEVSE charger's HTTP REST API every 10 seconds, exposes live charge state, session energy, current, voltage, and lifetime energy in Home Assistant, and lets you control max amperage and sleep/wake from the dashboard. Wallbox, Easee, and Zaptec are supported as variants - see [API differences](#api-differences) below.

## Features

- **Live polling** of charger state every 10 s via `http_request` + JSON parsing in a `text_sensor` lambda
- **Sensors**: charge state, session kWh, session time, current (A), voltage (V), real power (W), lifetime kWh
- **Control**: a `number` entity (0-32 A) that POSTs to `/override` to set the charging current, and a `switch` to sleep / wake the charger
- **Off-peak scheduling** automation (charges 23:00 to 07:00)
- **Solar-surplus charging** automation that only charges when grid solar export is above 1.5 kW (depends on `sensor.solar_inverter_power` from the solar-inverter-modbus example)
- **Fault notifications** for GFI, no-ground, stuck relay, and diode check failures
- **No cloud dependency** - all traffic stays on the LAN between the ESP32 and the charger

## Hardware

| Component | Purpose | Notes |
|-----------|---------|-------|
| ESP32 dev board | Host that polls the charger and bridges to HA | Any ESP32-WROOM-32 board works |
| WiFi link | Default network path | Same SSID as the charger |
| Optional: W5500 SPI Ethernet module | Stable wired link to the charging-station network | Recommended if the charger sits in a garage with weak WiFi |

The ESP32 does not connect to mains and never sees CP/PP pilot signals. All charge control happens over HTTP to the charger's own controller, which keeps it safe and certifiable.

## Supported chargers

| Charger | Protocol | Default in this example | Notes |
|---------|----------|------------------------|-------|
| [OpenEVSE](https://openevse.com) | HTTP REST + JSON, port 80 | yes | Fully open-source firmware. This example targets it directly. |
| [Wallbox](https://wallbox.com) | HTTP REST + JSON (myWallbox cloud) or local Modbus TCP (Commander 2 / Pulsar Plus) | variant | Local Modbus needs a JSON wrapper, see below |
| [Easee](https://easee.com) | HTTPS REST + JSON (Easee Cloud) | variant | No local API, all traffic must go through Easee's cloud |
| [Zaptec](https://zaptec.com) | HTTPS REST + JSON (Zaptec Cloud) | variant | Same as Easee, cloud-only |

For Easee and Zaptec, the polling logic is identical; only the URL, auth header, and JSON field names change.

## Installation

### 1. Find the charger on your LAN

For OpenEVSE, open its web UI and note the IP address (something like `192.168.1.50`). It is shown on the front of the device after first boot. Test the API with curl:

```bash
curl -u openevse: http://192.168.1.50/status
```

You should get a JSON blob back with `state`, `session_energy`, `amp`, `voltage`, and the other fields used by this example.

### 2. Configure secrets

```bash
cp secrets.yaml.example secrets.yaml
# edit secrets.yaml and fill in wifi_ssid, wifi_password, api_key, ota_password
```

### 3. Set the charger host in the YAML

Open `ev-charger-monitor.yaml` and edit the top substitutions:

```yaml
substitutions:
  charger_host: "http://192.168.1.50"
```

If you set a password in the OpenEVSE UI, also regenerate the `basic_auth_header` global:

```bash
printf 'openevse:yourpassword' | base64
```

Paste the output as the `initial_value` of `basic_auth_header` in the YAML.

### 4. Flash the ESP32

```bash
esphome run ev-charger-monitor.yaml
```

### 5. Verify in Home Assistant

After flashing:
- HA discovers the new device via the ESPHome integration
- `sensor.ev_charger_monitor_state` shows `not_connected`, `connected`, or `charging`
- Plug the car in - the state should change within one poll cycle (10 s)
- Adjust `number.ev_charger_monitor_max_amps` and watch the charging current update

### 6. Add automations + dashboard

Copy the contents of `automations.yaml` and `dashboard.yaml` to your Home Assistant configuration.

## API differences

### OpenEVSE (default)

`GET /status` returns a JSON object with these fields (relevant subset):

| Field | Type | Meaning |
|-------|------|---------|
| `state` | int | 1 not connected, 2 connected, 3 charging, 4-8 faults, 254 sleeping |
| `session_energy` | int | Wh in the current session |
| `session_elapsed` | int | seconds since session start |
| `amp` | int | charging current in milliamps |
| `voltage` | int | volts |
| `total_energy` | float | lifetime kWh |

`POST /override` accepts a JSON body. To set current: `{"charge_current":16}`. To sleep: `{"state":"disabled"}`. To wake: `{"state":"active"}`.

### Wallbox (local Modbus on Commander 2 / Pulsar Plus)

Local Modbus TCP on port 502. The relevant holding registers:

| Register | Meaning |
|----------|---------|
| 0x0000 | charger state (0 ready, 1 charging, 2 connected paused, 4 error) |
| 0x0102 | max current setpoint (A) |
| 0x0212 | session energy (Wh) |

To use Wallbox, replace the `http_request` polling block with a `modbus_controller:` and `modbus_controller.sensor` entries pointing at these registers. Aurora's `modbus_controller` reference covers the wiring.

### Easee and Zaptec (cloud only)

Both use HTTPS REST APIs that require OAuth tokens (Easee bearer token, Zaptec installation key). Change `charger_host` to the cloud URL and add a `Bearer` token to the `Authorization` header. The polling lambda then parses fields named differently:

| Field | OpenEVSE | Easee | Zaptec |
|-------|----------|-------|--------|
| state | `state` | `chargerOpMode` | `OperatingMode` |
| session energy | `session_energy` | `sessionEnergy` | `TotalChargePower` |
| current | `amp` | `inCurrent` | `CurrentPhase1` |
| voltage | `voltage` | `inVoltage` | `VoltagePhase1` |

Update the `find("\"state\":")` offsets in the YAML's lambdas to match.

## Troubleshooting

| Issue | Likely cause | Fix |
|-------|-------------|-----|
| All sensors stay `unknown` | ESP32 cannot reach the charger | Ping the IP from the same network, check `charger_host` in the YAML |
| Sensors update but values are wrong | JSON field offsets do not match your charger firmware | Capture `curl http://CHARGER/status` and compare field names to the lambdas |
| Setting max amps does nothing | OpenEVSE has the charge schedule overriding manual control | Disable the schedule in the OpenEVSE UI before letting HA control current |
| HTTP 401 on every poll | Auth header is wrong | Regenerate `basic_auth_header` with `printf 'user:pass' | base64` |
| Charging keeps stopping under solar control | Surplus is right at the 1.5 kW threshold | Raise the `for:` duration in the solar-stop automation to avoid flapping |
