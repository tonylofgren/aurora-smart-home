# DEEP-Mode Snapshot Demo (CO2 Air Quality)

> *Generated by [aurora@aurora-smart-home (aurora skill)](https://github.com/tonylofgren/aurora-smart-home)*

A worked example of Aurora's DEEP-mode hand-off protocol: three specialists building one project through a single shared snapshot, `aurora-project.json`. It exists to make the protocol concrete and to give `test_deep_mode_snapshot_replay.py` something real to replay.

## The flow

1. **Volt** picks the board (`esp32-c3-mini`), the sensor (`scd40`), allocates GPIO (SDA 6 / SCL 7), writes `esphome/co2-monitor.yaml`, runs the hardware validators, and records `validation_results.volt`.
2. **Sage** reads Volt's entity IDs, writes the CO2 automation to `packages/co2_monitor.yaml`, and records `validation_results.sage`.
3. **Iris** reads the entities, builds `dashboards/air-quality.yaml`, and records `validation_results.iris`. Iris briefly raised a `conflict_log` entry because the humidity entity was not yet listed; Sage confirmed it and Iris proceeded. The entry is shown resolved.

## What the snapshot demonstrates

- **Per-field ownership:** Volt owns `selected_board`, `selected_components`, `gpio_allocation`, `esphome_filename`; Sage and Iris own their `ha_yaml_files`; each agent owns only `validation_results.<self>`. See `aurora/references/handoff/_protocol.md`.
- **Append-only progress:** `agents_completed` is the ordered pipeline; `agents_pending` drains to empty.
- **Conflict escape hatch:** a raised-then-resolved `conflict_log` entry instead of an agent silently overwriting a peer's field.
- **Referential integrity:** every path in `esphome_filename` and `ha_yaml_files` exists in this folder, every entity ID is well-formed, the board and component IDs match profiles under `aurora/references/`.
