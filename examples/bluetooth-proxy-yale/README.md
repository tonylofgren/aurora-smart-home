# Bluetooth Proxy for Yale / August Locks

> *Generated by [aurora@aurora-smart-home (esphome skill)](https://github.com/tonylofgren/aurora-smart-home)*

A Bluetooth proxy build that benefits from three concrete fixes/features
in ESPHome 2026.5.0:

1. **BLE/WiFi coex fix** - the `status=133` GATT failure that Yale and
   August lock owners hit for years under heavy WiFi load is resolved.
   The proxy now holds Bluetooth priority for the full duration of
   active connections.
2. **`esp32_ble: use_psram`** - frees about 40 kB of internal RAM by
   moving the Bluedroid stack into SPIRAM. Critical on busy proxies.
3. **`esp32: watchdog_timeout: 15s`** - gives long GATT operations
   more breathing room than the default 5 s.

Pair this with a WROVER or S3-with-PSRAM board for the best results.

## Features

- Active BLE connection support (works for locks, thermostats,
  Bluetooth-only sensors)
- Service cache so reconnects are sub-second
- Wide scan window so slow-advertising devices are picked up
- WiFi signal sensor at the proxy location (placement-critical)
- Status, uptime, IP, and ESPHome version exposed to HA for
  diagnostics

## Hardware List

| Component | Purpose | Notes |
|-----------|---------|-------|
| ESP32-WROVER dev board | Host with PSRAM | Best price/perf for a busy proxy. ESP32-S3-N16R8 also works. |
| 5 V USB-C power supply | Power | Always-on. Use a quality supply; bad cables cause WiFi dropouts. |
| External 2.4 GHz antenna (optional) | Range | Useful if the proxy is in a corner |
| Small case | Looks better than bare PCB | Many community-designed 3D-printable cases for WROVER exist |

**Boards that work but are not recommended:** plain ESP32-WROOM (no PSRAM,
falls back to internal-RAM Bluedroid, may run out of heap when proxying
3+ active devices), ESP32-C3 (single core, no PSRAM, marginal).

## Placement

The proxy needs:
- Solid WiFi (think 2 walls max from the AP, RSSI better than -65 dBm)
- Line-of-sight to the BLE devices you're proxying (2-3 walls max for a
  lock; closer for low-power BLE thermometers)

A typical layout: one proxy per floor, placed centrally. Apartments
usually need only one.

## Installation

### 1. Flash

```bash
esphome run bluetooth-proxy-yale.yaml
```

### 2. Adopt in HA

After flashing, the device shows up under Settings > Devices &
Services > ESPHome with a button to "Proxy Bluetooth Devices". Click
that.

### 3. Add the lock to HA

For a Yale or August lock:
1. Settings > Devices & Services > Add Integration > Yale (or August)
2. Sign in with your Yale/August account
3. The integration finds the lock through your new ESP proxy

For a generic BLE-only device, the integration is device-specific (e.g.
Govee thermometers go through the Govee BLE integration).

### 4. Verify proxied connection

Check the WiFi signal sensor in HA. If it's consistently below -70 dBm,
move the proxy. The 2026.5.0 coex fix helps WiFi-vs-BT contention but
cannot rescue a poor WiFi signal.

## Configuration

### Multiple proxies

For larger houses, flash multiple proxies. Each needs a unique
`devicename`. HA automatically routes proxy traffic to whichever proxy
hears the target device best.

```yaml
substitutions:
  devicename: "bt-proxy-upstairs"        # change per device
  long_devicename: "BT Proxy Upstairs"
```

### Tuning the scan parameters

Defaults are tuned for active connections. If you have lots of passive
BLE beacons (RuuviTags, Mi Flora, etc.) and want faster discovery:

```yaml
esp32_ble_tracker:
  scan_parameters:
    interval: 320ms
    window: 320ms
    active: false       # passive scanning is plenty for beacons
```

But note: passive scanning + active connections (locks) on the same
proxy can fight. Run two proxies if you need both at high quality.

## Troubleshooting

| Issue | Likely cause | Fix |
|-------|-------------|-----|
| Lock unlock takes 19 s and then fails with status 133 | Old ESPHome (pre-2026.5.0), or a different proxy in the house running old firmware | Upgrade all proxies to 2026.5.0+. The fix only works when the proxy that actually connects is patched. |
| WiFi throughput drops while the lock is connected | Expected behavior of the coex fix | Either accept it (lock connections are brief) or run a dedicated proxy in a less-WiFi-critical room |
| Out-of-memory crash | Plain ESP32 without PSRAM + busy proxy | Switch to ESP32-WROVER or ESP32-S3 with PSRAM. Enable `use_psram`. |
| Proxy goes offline a few times a day | Power supply or WiFi power-save | Replace the USB cable/PSU. We set `power_save_mode: NONE`; confirm your YAML matches. |
| Lock works at first, then needs HA restart to unlock again | Old firmware bug | Upgrade to 2026.5.0. Also ensure `cache_services: true` is set. |
| Cannot find specific device | Out of range OR active scanning required | Move proxy closer or set `scan_parameters: active: true`. |

## Why 2026.5.0 matters here

Before this release, Bluetooth Proxy users with active BLE connections
under heavy WiFi load reported `status=133` (decimal) or `0x85` (hex)
GATT failures. The proxy would attempt the unlock, time out at ~19 s,
and fail. The cause was the BLE/WiFi coex policy releasing BT priority
as soon as the GATT handshake settled, even though the rest of the
GATT operation still needed BT airtime.

The 2026.5.0 fix holds BT priority for the entire lifetime of an
active connection (CONNECTED + ESTABLISHED states). Real-world test
against a Yale BETA211123 lock with the patched proxy: clean unlock
in 228 ms vs. timeout at 19 s before.

**Side effect:** during an active connection, WiFi throughput on the
proxy is reduced. That's the correct tradeoff for lock reliability;
if you don't want it, run a dedicated proxy in a less-WiFi-critical
location.
