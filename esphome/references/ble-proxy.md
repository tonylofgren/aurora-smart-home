# BLE Proxy

Guide for configuring ESP32 as a Bluetooth Low Energy (BLE) proxy for Home Assistant.

## Overview

BLE Proxy extends Home Assistant's Bluetooth range by placing ESP32 devices strategically throughout the home. Each proxy forwards BLE advertisements to Home Assistant via the ESPHome API.

## Use Cases

- **Xiaomi/Mijia sensors** - Temperature, humidity, plants
- **iBeacons** - Presence detection
- **Tile/AirTag** - Tracking
- **Bluetooth thermometers** - Kitchen, grill, pool
- **BLE locks** - Smart locks

## Hardware Requirements

| Chip | BLE Proxy | Active Connections | Recommendation |
|------|-----------|-------------------|----------------|
| ESP32 | ✅ Yes | 3 | Standard, cheap |
| ESP32-S3 | ✅ Yes | 9 | Better performance |
| ESP32-C3 | ✅ Yes | 3 | Compact |
| ESP32-C6 | ✅ Yes | 3 | Thread + BLE |
| ESP8266 | ❌ No | - | Lacks BLE |

## Minimal BLE Proxy

```yaml
# Generated by aurora@aurora-smart-home (esphome skill) v1.7.9
# https://github.com/tonylofgren/aurora-smart-home

esphome:
  name: ble-proxy-living-room
  friendly_name: "BLE Proxy Living Room"

esp32:
  board: esp32dev
  framework:
    type: esp-idf  # Recommended for BLE

wifi:
  ssid: !secret wifi_ssid
  password: !secret wifi_password

api:

ota:
  platform: esphome

logger:

# BLE Proxy - Forwards to Home Assistant
esp32_ble_tracker:
  scan_parameters:
    active: true
    interval: 1100ms
    window: 1100ms

bluetooth_proxy:
  active: true  # Allows active connections
```

  active: true

## BLE Presence Detection

Detect presence via BLE devices (phones, smartwatches):

```yaml
# Generated by aurora@aurora-smart-home (esphome skill) v1.7.9
# https://github.com/tonylofgren/aurora-smart-home

esphome:
  name: ble-presence
  friendly_name: "BLE Presence"

esp32:
  board: esp32dev
  framework:
    type: esp-idf

wifi:
  ssid: !secret wifi_ssid
  password: !secret wifi_password

api:
ota:
  platform: esphome
logger:

esp32_ble_tracker:
  scan_parameters:
    active: false  # Passive scanning is sufficient
    interval: 1100ms
    window: 1100ms

bluetooth_proxy:
  active: false  # Tracking only

# iBeacon presence
binary_sensor:
  - platform: ble_presence
    ibeacon_uuid: "YOUR-IBEACON-UUID"
    name: "iPhone Home"

  # MAC-based (less reliable)
  - platform: ble_presence
    mac_address: "AA:BB:CC:DD:EE:FF"
    name: "Smartwatch Home"
    timeout: 5min

# RSSI for distance
sensor:
  - platform: ble_rssi
    mac_address: "AA:BB:CC:DD:EE:FF"
    name: "Phone RSSI"
    filters:
      - exponential_moving_average:
          alpha: 0.3
          send_every: 1
```

## ESP-IDF vs Arduino Framework

| Feature | Arduino | ESP-IDF |
|---------|---------|---------|
| BLE Stability | Good | Very good |
| Active Connections | 3 | 3-9 (chip-dependent) |
| Memory Usage | Higher | Lower |
| Compilation Time | Faster | Slower |
| Recommendation | Simple projects | BLE Proxy |

**Use ESP-IDF for BLE Proxy:**
```yaml
esp32:
  board: esp32dev
  framework:
    type: esp-idf
```

## Scan Parameters

| Parameter | Value | Description |
|-----------|-------|-------------|
| `interval` | 1100ms | Time between scans |
| `window` | 1100ms | Time per scan |
| `active` | true/false | Active querying |
| `continuous` | true/false | Continuous scanning |

**Rules of thumb:**
- `window` ≤ `interval`
- `window = interval` for maximum coverage
- `active: true` for devices that require it
- `continuous: true` for presence detection

## Troubleshooting

### Device not visible

1. **Check MAC address** - Verify with nRF Connect app
2. **Enable active scanning** - Some devices require it
3. **Increase window** - Longer scanning period
4. **Check range** - BLE has ~10m range

### Unstable connection

1. **Use ESP-IDF** framework
2. **Reduce number of active connections**
3. **Increase scan interval**
4. **Check WiFi interference** (2.4GHz)
5. **Upgrade to ESPHome 2026.5.0+** if you see `status=0x85` / `status=133` GATT failures (see below)

### status=0x85 / 133 GATT failures (fixed in 2026.5.0)

A long-standing class of `status=0x85` (decimal 133) errors hit Bluetooth Proxy users under sustained WiFi traffic. Most visibly affected: Yale and August lock owners.

**Symptom:** under heavy WiFi load, GATT operations time out after ~19 s with `status=133`. Same lock works fine when WiFi is idle.

**Fix in 2026.5.0:** the BLE tracker now holds `ESP_COEX_PREFER_BT` for the full lifetime of any active BLE connection (CONNECTED + ESTABLISHED states), instead of reverting to balanced coex as soon as the handshake settled. Production validation against a Yale BETA211123 lock with host BlueZ disabled: clean unlock via the patched proxy in ~228 ms vs. timeout at 19 s before.

**Side effect:** configs holding long-lived BLE connections (for example a `ble_client` with a persistent sensor) may see lower WiFi throughput while the connection is up. This is expected. If it materially affects your workload, open an issue.

### High CPU/memory usage

```yaml
esp32_ble_tracker:
  scan_parameters:
    active: false  # Passive if possible
    interval: 2000ms  # Increase interval
    window: 1000ms

esp32_ble:
  use_psram: true   # 2026.5.0+: directs Bluedroid to SPIRAM (~40 kB internal RAM freed)
```

The `esp32_ble: use_psram: true` option (new in 2026.5.0) is only useful on PSRAM-equipped boards. It moves the Bluedroid stack into SPIRAM, freeing roughly 40 kB of internal RAM for your code/buffers.

## Home Assistant Integration

BLE Proxy integrates automatically:

1. **Install ESPHome integration** in HA
2. **Adopt the device** when it is discovered
3. **BLE devices appear** under Integrations - Bluetooth

**No extra configuration needed** - Home Assistant sees all BLE devices via the proxy.
